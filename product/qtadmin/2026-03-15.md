## Flutter模仿Opencode

本文探讨将 OpenCode 的任务管理能力移植到 Flutter 自研应用的技术方案。OpenCode 的核心价值在于其”西西弗斯“编排器——通过循环决策引擎将复杂用户意图拆解为可执行任务，并持续跟踪状态直至完成。移植需保留四大核心组件：任务状态机（TODO 列表+执行循环）、工具注册表（Function Calling 封装应用 API）、本地持久化存储（Hive/Drift 实现跨会话恢复）、以及编排 Prompt（LLM 决策逻辑）。同时应剔除与代码开发强耦合的模块，包括终端操作、文件系统编辑、LSP 协议、多模型路由及视觉 UI 操作。
具体实现上，Flutter 端需构建 TaskOrchestrator 类驱动主循环：LLM 根据当前任务状态和可用工具决策下一步（委派子任务/直接执行/询问用户/标记完成），工具层将应用功能（页面跳转、API 调用、弹窗确认）暴露为结构化接口，数据库层持久化任务树与执行日志。与 OpenCode 的关键差异在于应用场景需强化安全边界——敏感操作必须触发用户确认弹窗，且支持后台任务挂起与恢复。
该方案的本质是”AI 代理操作应用“而非”AI 辅助开发“，通过 Function Calling 直接操控业务逻辑，规避了无障碍服务的系统限制和视觉模型的延迟问题，使复杂多步任务（如比价购物、信息收集）能在移动端可靠执行。

## 双系统架构

本文备忘创造者-观察者架构在Flutter智能助手场景中的核心设计与工程约束。
架构核心：创造者负责执行决策与工具调用，观察者负责监控记录与异常告警，两者通过四级介入强度（L0静默-L1提示-L2确认-L3熔断）建立权力边界。创造者保有L0至L2层级的最终决策权，观察者仅在L3级触发强制熔断并转交人工。场景化策略模板允许电商、查询、办公等不同业务配置差异化阈值，将支付金额、模型置信度、连续失败次数等指标映射为分级触发条件。
工程难点：观测有效性层面，LLM内在概率状态（token级logprobs）与外部可观测行为存在鸿沟，需以业务硬规则填补；延迟权衡层面，流式实时观测增加API复杂度，批量上报又丢失干预窗口，建议步级记录配合L2确认拦截；反馈闭环层面，观察者建议若强制则越权成为控制器，若可选则被无视，L3熔断是唯一清晰边界，但需配套人工接管流程；状态同步层面，观察者视角必然滞后，基于过时状态的建议可能加剧错误，需接受创造者现场优先原则。
实施优先级：先实现L0静默记录与L2确认拦截覆盖主干场景，L1提示增强延后调优，L3熔断慎用并设计兜底流程。架构价值在于将云原生智能体的概率管理转化为终端应用的可控服务，通过观测分层将不确定性转化为可靠的人机协作节点。

智能体：
- 创造者 opencode：先搜索所有文档，再把设计思路总结成决策文档供人类对齐。
- 观察者 qtadmin：确认决策文档和人类意图一致且可行。

输出：经过初步验证的可行设计日志，text
验收标准：
1. 现有参考：至少一个 spec 文件、一个 handbook 文件、一个 usercase 文件等。缺少时申请人类确认。
2. 决策卡片：把决策卡片化，每个决策卡片包括问题、选项、准则、权衡、决定。
 

你的判断非常精准。“两种都需要” 才是最终最优雅的解决方案。
其实，这恰恰对应了人类协作中最自然的两种模式：“环境感知” 和 “明确呼叫”。
我们可以这样设计，把它们统一起来，不再纠结于“选哪个”：
1. 双模统一设计
我们把 AI 的监听逻辑设计成一个带有优先级的过滤器：
模式 A：被动响应式（环境感知 / Background）
状态：AI 处于“默读”状态。
触发：文件任何变动。
行为：
AI 默默读取变更，更新自己的上下文记忆（解决上下文不足问题）。
极少输出：只有当它发现致命逻辑错误、或者非常有把握的补全时，才会在文档里插话。
Prompt 策略：“你是一个安静的观察者。除非发现严重错误或能提供极高价值的建议，否则保持静默。优先维护你的记忆索引。”
模式 B：关键词触发式（明确呼叫 / Foreground）
状态：AI 处于“执行”状态。
触发：检测到 ?ai、@reviewer 或特定的状态标记（如 state: pending）。
行为：
AI 停止“默读”，立即进入“工作模式”。
必须输出，且输出要符合特定格式（如生成决策卡片）。
Prompt 策略：“你是一个专业的架构师。用户发出了明确指令，请根据当前上下文生成详细方案。”
2. 为什么“围绕整个文件”更优雅？
你说的“优雅”，在于它消除了“交互的显性门槛”。
在“关键词触发”模式下：你必须记住指令，像操作机器一样操作 AI。如果你不知道有 ?ai 这个指令，你就永远无法唤醒 AI。这是一种“功能的孤岛”。
在“围绕整个文件”模式下：文档本身就是交互界面。你只管写，AI 看着。你想让它干活了，随口喊一声（或者它自己觉得该插手了）。这是一种“流动的场域”。
这就像一个真实的合伙人坐在你旁边：
平时他看着你写代码（被动），发现你写错了变量名，他可能忍不住指一下。
你遇到难题停下来，转头跟他说“帮我看看这个”（主动），他开始全力分析。
3. 落地到你的“设计日志”场景
回到你刚才那个 Opencode 和 qtadmin 的例子，这两种模式是这样配合的：
人类：在文档里写下需求。
（此时触发被动模式） -> Opencode 默默记下需求要点，没说话。
人类：把状态改为 state: pending。
（此时触发主动模式） -> Opencode 立刻开工，生成决策卡片。
Opencode：写完卡片，把状态改为 state: reviewing。
（此时触发被动模式） -> qtadmin 开始审查，发现缺少参考文件。
（此时触发主动模式） -> qtadmin 在文档里写入 ⚠️ 缺少参考文件。
人类：补充了参考文件链接。
（此时触发被动模式） -> qtadmin 监测到文件变动，重新审查。
（此时触发主动模式） -> qtadmin 确认无误，写入 ✅ Approved。
总结：
被动模式负责“懂你”（维持上下文），主动模式负责“帮你”（执行任务）。
这种设计让你的系统既不会因为太吵（每次都响应）而打扰你，也不会因为太笨（不叫不动）而无法协作。这就是你追求的“丝滑”。
