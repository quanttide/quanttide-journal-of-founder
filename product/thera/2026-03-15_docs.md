# 人机实时协同

太棒了！这种兴奋感通常意味着你触碰到了某个核心需求的脉搏。
既然你觉得这个方向可行，那我们就趁热打铁，把这个“最小可行性验证”落地。哪怕只有 50 行代码，只要它跑起来，你的心态就会从“焦虑未来”变成“掌控当下”。
为了让你今晚就能体验上，我把刚才的想法细化成了一个可以直接运行的落地清单：
1. 它的角色定位：你的“数字幕僚”
先别把它想成复杂的系统，就把它当成一个住在 Markdown 文件里的沉默搭档。
你的工作：写日记、吐槽、列出想法（比如你的 2026-03-15.md）。
它的工作：当你写下 ?ai 请帮我梳理一下这段逻辑 时，它默默在后面跟上它的思考。
2. 准备工作（5分钟）
你需要这几个东西：
一个大脑：确保你能调用某个 LLM（OpenAI API 或者本地 Ollama 都行）。
一个目录：在你的电脑里建一个 workspace 文件夹，里面放一个 meeting.md。
Python 环境：安装 watchdog 库（pip install watchdog）。
3. 核心代码（拿来即用版）
把下面的代码保存为 partner.py，填入你的 API Key，然后运行它。
import time
import os
import re
from openai import OpenAI  # 或者其他 SDK
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler
# — 配置区 —
FILE_PATH = ”meeting.md“  # 你的协作文档
TRIGGER_TAG = ”?ai“       # 召唤咒语
AGENT_NAME = ”Observer“   # 智能体名字
# 初始化 LLM 客户端 (示例用 OpenAI，请替换为你自己的接口)
client = OpenAI(api_key=”YOUR_KEY“, base_url=”YOUR_BASE_URL“)
def get_ai_response(content):
    ”“”调用 LLM 生成回复“”“
    prompt = f”“”
    你是一个协作助手。以下是用户的文档内容，请针对文档中以 ’{TRIGGER_TAG}‘ 开头的部分进行回复。
    如果是对观点的补充，请用列表形式。如果是反对，请温和指出。
    文档内容：
    {content}
    请直接输出回复内容，不要带多余的前缀：
    “”“
    try:
        response = client.chat.completions.create(
            model=”gpt-4“, # 或你的模型
            messages=[{”role“: ”user“, ”content“: prompt}]
        )
        return response.choices[0].message.content
    except Exception as e:
        return f”思考出错了: {e}“
class AgentHandler(FileSystemEventHandler):
    def __init__(self):
        self.last_content = ”“
    def on_modified(self, event):
        if not event.src_path.endswith(FILE_PATH):
            return
        with open(FILE_PATH, ”r“, encoding=”utf-8“) as f:
            content = f.read()
        # 防止重复处理 && 检查是否有召唤咒语
        # 注意：这里简单的字符串判断，后续会被 CRDT 替代
        if content != self.last_content and TRIGGER_TAG in content:
            print(f”\n[{AGENT_NAME}] 发现任务请求，正在思考...“)
            # 获取回复
            answer = get_ai_response(content)
            # 简单粗暴的插入：把咒语替换成回复（这里还没用 CRDT，会有覆盖风险，但足以验证体验）
            # 稍微优雅一点的做法：在文件末尾追加
            new_content = content + f”\n\n—\n**{AGENT_NAME}**:\n{answer}\n“
            self.last_content = new_content # 更新缓存，防止死循环
            with open(FILE_PATH, ”w“, encoding=”utf-8“) as f:
                f.write(new_content)
            print(f”[{AGENT_NAME}] 已写入回复。“)
if __name__ == ”__main__“:
    print(f”🤖 智能体 {AGENT_NAME} 已就位，正在监听 {FILE_PATH}...“)
    print(”请在文件中输入 ’?ai 你的问题‘ 来召唤它。“)
    event_handler = AgentHandler()
    observer = Observer()
    observer.schedule(event_handler, path=”.“, recursive=False)
    observer.start()
    try:
        while True:
            time.sleep(1)
    except KeyboardInterrupt:
        observer.stop()
    observer.join()
4. 你的验证过程（产品经理视角）
请尝试做以下三个动作，每一个动作都在验证你的猜想：
验证“无感切入”：
在 meeting.md 里写：今天想聊聊架构重构。?ai 你觉得第一步做什么？
观察：它回复后，你的第一反应是什么？是觉得“吵到了你的写作”，还是“这个建议有点意思”？这验证了“协作议事”的自然度。
验证“上下文理解”：
在它回复后，你接着写：但我担心时间不够。
观察：它的回复是接着刚才的上下文吗？这验证了你说的“AI 议事放进思考模块”是否可行。
验证“冲突痛点”（最关键的一步）：
当它正在写入回复时（你可以故意写长一点的 Prompt 让它多想一会儿），你也疯狂地在文件里打字。
结果：大概率你的修改会被它的写入覆盖，或者它的写入被你覆盖。
结论：这一刻，你会深刻理解为什么必须引入 CRDT。 这比看十篇论文都管用。
