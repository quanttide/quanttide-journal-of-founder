# 2026-03-12

今天 Gedankenexperiment（思想实验）比较多，记录一下。

## 基础设施与容器化

既然有写示例的需求，就可以把需求封装成容器，类似小型 k8s 或函数集群，可以快速自动化各种日常工作流程。主要难点还是外化，舒适的外化能带来更舒适的编码体验。

封装基础设施是个非常好的积累组件重组流程的起点，比如对象存储、函数等。

另外，用个人或公司账号回顾和腾讯云的博弈，可以作为公司发展历程的一部分。虽然心里有怨气不吐不快，但太零散了不知道从哪写起。这是典型的专项写作需求，需要完整的叙事线。

## 知识工作模块

有了一个通用的知识工作模块以后，就可以打通飞书知识库和 GitHub 仓库，逐渐建立高效的工作流。工程思路可以从连通这两个平台开始。

之前在个人外脑里做过的飞书到 GitHub 的方案正是目前需要的。腾讯云的通用连接器也是一个可以集成的方案。

## 文档驱动开发

在思考用文档生成代码，然后用代码反过来循环生成文档的思路是否可行。这种思路有可能让整个迭代更高效。因为 AI 写的代码容易越写越乱，而代码是从文档来的，用文档驱动不就可以了么？

把 examples 当作实验室用就比较合适。example 的状态是从混沌到清楚，然后到被精简成可以直接调用库就能解决。这需要对 examples 的状态有更清晰的定义。

## 编程任务分解

建立了一个流程：能识别出是产品需求就直接往 PRD 写；知道是设计实现就往开发者文档写；工作计划文档留给 AI 维护，主要看工作计划是否清楚。

核心是：PRD -> 开发者文档 -> AI 维护的 plan 文档 -> 代码 -> QA 文档。

plan 文档比项目管理文档或产品设计文档更 specific，更临时，只回答 AI 要怎么干事。随着开发者文档的细化，plan 文档可能逐渐不需要。

人工维护的 specification 比手动维护的更精密，用 line 兼容现有规范很合理。关键是要先做出能准确生成代码的文档，才知道文档要怎么规范。

## 移动端工作流

在电脑上用命令行完成模块化后，就可以在手机上反复对某个模块进行迭代，再回到电脑做正式开发。这可能提高协同效率——有想法时随时随地写一下。

如果自己的平台能在手机上完成分解工作的状态和交互，理论上效率可以提高很多。这隐含了把编程分解成更准确任务的想法。

## 文档体系调整

手机端写东西越来越频繁，发现用 essay 的形式比 handbook 顺手多了。essay 更适合倾诉和捕捉灵感，不需要太正式；handbook 还是应该作为团队的正式参考文档，保持结构化和权威性。

bylaws 也可以开始动笔了，规章制度需要给团队清晰的示范。

## 工作流与技术配置

一直在想怎么打通内部知识库和外部代码仓库。理想的工作流是：在本地把飞书知识库的数据同步下来，同时拉取 GitHub 仓库，然后半自动化编辑内容，最后让 AI 帮忙自动提交 PR。

模型还是更习惯 DeepSeek，得去阿里云百炼重新配置参数。顺便把百炼加到收藏夹，也想试试用命令行直接配置，结合网页版检查，对 opencode 的能力边界有更直观的了解。

## 标准与协议

之前把 AI 相关协议放在云计算标准里，现在觉得不太合适。AI 的社区标准已经成熟了，不需要自己再造轮子。准备挪到基础设施标准里，这样标准可以更纯粹地兼容外部协议，屏蔽供应商细节，增强通用性。

关于是否使用的终极标准，就是看有没有透明公开共识的标准，即符合基础设施的定义。这样才能让不同的数据在不同软件流转，这是建立组织所有权的关键环节之一。

开放格式的标准才是企业真正的资产。数字时代的所有权正在发生巨大的变化。

## 想法整合

过去吸收的各种想法正在快速整合成新体系，有个新范式正在脑子里长出来，只是还没有落地。对未来的想象更明确了，在新方法论下可以逐渐分层，让组织内不同人处理不同的层次。

如果能把核心算法拆出来标准化，复用就会更容易。本质上是用更多的排列组合来实现创新想法。这里有核心的「自我」，AI 一时半会还不容易学明白，而核心想法恰恰是希望它学明白这个，并且希望每个人和每个组织都可以有自己的元方法。

## RuyiX 评估

RuyiX 利用大模型的能力设置了元编程方案，巧妙在可以转成编程语言作为中介。用模糊的命令生成伪代码再执行也是一样的，甚至更透明。他们不怎么开放，暂时不合作了，吸收这个思路就可以。

## 开源与认知安全

这种实践的主要风险是担心万一火了以后面对更多争议。但是目前来看概率不高，真到那个时候也应该会更有经验。最差的结果就是取消开源，也不会有什么本质影响，或者搭一个闭源层来处理。

开源主要是容易暴露认知上的局限，可能会被有心人利用。虽然比较难，但它实际上回答的是：如果我们要去搭面向认知的基础设施，认知被外化后暴露风险会增加，攻击难度也会增加，那么对于认知安全就有了新要求，会成为认知工程的一部分。

智能体的安全和大模型在语义上的攻击是一个问题，加上人之后也还是一个问题。会对隐私保护、安全防护有新的要求。

## AI Native 应用

AI Native 应用的关键是用各种规则引擎进行意图分类和状态管理。人类已经有所有能够治理草台班子的规则引擎，只是根据 AI 的情况选择合适的引擎是需要专门探索的。

## 效率工具

以前想到要写备忘录上思考怎么分类，变成批处理以后这个事情被延后了、集中了、提高了效率。这样收集起来会更没有负担。

语音输入是比较重要的基本单元，得准备一个方案备用。

验证了思想：先看数据最后看智能是对的。调提示词也是不小的坑，需要让 AI 慢慢调。

cli-provider-studio 是正确的路线：要先验证 AI 操作 cli 可行，然后前后端分离，最后给人类检查。

这个工程思路有卡片笔记的感觉。系统和卡片笔记系统比较接近，并且效果更好，主要兼容了所有的文体，把各种表达方法都集中到了体系里。
